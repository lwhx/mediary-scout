# 115 扫码登录实现简报（2026-06-12）

> 目标：web 原生扫码连接 115，服务端拿到并持久化 cookie（UID/CID/SEID/KID），替代 .env 手填。
> 流程已对照一手源码验证：SheltonZhu/115driver（Go，AList 后端）`pkg/driver/qrcode.go` 与
> ChenyangGao/p115client（Python）`client.py`。无需 AppID/签名——sign 由第一步服务端下发。

## 三步流程

### 1. 取二维码 token

```
GET https://qrcodeapi.115.com/api/1.0/web/1.0/token/
```

无参数。此处路径里的 `web` **不**决定绑定的客户端类型（绑定发生在第 3 步），固定用 `web` 即可。

响应 `data`: `{ uid, time, sign, qrcode }`（`state==1` 成功）。`qrcode` 是二维码内容字符串
（缺省可用 `https://115.com/scan/dg-${uid}` 兜底，p115client 即如此）。

### 2. 渲染二维码

推荐前端用 `qrcode` npm 包渲染 `data.qrcode` 字符串（全自托管）；
或代理官方 PNG：`GET https://qrcodeapi.115.com/api/1.0/web/1.0/qrcode?uid={uid}`。
用户用 **115生活 App** 扫码并确认。

### 3. 轮询扫码状态（长轮询）

```
GET https://qrcodeapi.115.com/get/status/?uid={uid}&time={time}&sign={sign}&_={Date.now()}
```

`data.status`：`0` 未扫 / `1` 已扫待确认 / `2` 已确认（→第 4 步）/ `-1` 过期 / `-2` 手机端取消。

该端点是长轮询（挂住 ~30s 或状态变化才返回）。Node 侧：fetch 超时设 ~35s，响应后立刻重发，
总 deadline ~5 分钟后视为过期给"重新生成"按钮。**长轮询超时时 `data` 可能没有 `status` 字段
——按"仍在等待"处理，不是错误**。Next.js 架构：`{uid,time,sign}` 存服务端，浏览器反复调
`/api/115/qrcode/status`（每请求一轮长轮询）。

### 4. 换 cookie（仅 status==2 后）

```
POST https://passportapi.115.com/app/1.0/{app}/1.0/login/qrcode/
Content-Type: application/x-www-form-urlencoded
account={uid}&app={app}
```

响应 `data.cookie`: `{ UID, CID, SEID, KID }` + `user_id/user_name/is_vip`。
**四个字段全存**（KID 较新可能为空；UID/CID/SEID 必须有），序列化为
`UID=...; CID=...; SEID=...; KID=...`。UID 值本身编码客户端类型：
`{user_id}_{ssoent}_{timestamp}`，ssoent `A1`=web、`R1`=wechatmini、`R2`=alipaymini、`I1`=tv。

## app 参数选型：默认 `alipaymini`

115 每客户端类型（ssoent）只允许一个活跃会话，以类型 X 登录会踢掉已有 X 会话
（即 cookie-manager 扩展那句"如果所选客户端类型已登录，该设备可能被登出"）。

- `web` 最差：浏览器登录 115.com 即失效，且 `app="web"` 最易触发"IP登录异常"风控
  （锁到北京时间次日凌晨，p115client 文档原话）。
- **`alipaymini` / `wechatmini` / `tv`** 用户几乎不会自然登录，cookie 实质长效
  （社区"理论上永不过期"）；p115client 默认就是 `alipaymini`。
- `windows/mac/linux` 已废弃（115driver 注释掉，桌面客户端已下架）。

**推荐**：默认 `alipaymini`，下拉提供 `alipaymini/wechatmini/tv/android/ios/web`，
附"所选类型已有会话会被登出"的警告。

## 坑

1. **互踢**：默认小程序类型避免惊吓用户。
2. **风控"IP登录异常"**：服务器 IP 反复登录会被锁到次日零点。缓解：登录一次→cookie 持久化
   （数据库，不是 .env）→仅在 API 返回鉴权失败时才重新走扫码；绝不自动循环重登。
3. **二维码 ~5 分钟过期**：status `-1` 从第 1 步重来。
4. **UA 一致性**：扫码请求带正常浏览器 UA + `Referer: https://115.com/`，且**之后所有带
   cookie 的 API 调用复用同一 UA**（115 风控绑定 UA 一致性；参考实现连 UA 一起存）。
5. **有效性检查**：`GET https://passportapi.115.com/app/1.0/web/1.0/check/sso`（带 cookie）
   可廉价验活（115driver `ApiLoginCheck`）。
6. **未来备选**：115 开放平台 PKCE 设备码流（`/open/authDeviceCode` → 轮询 →
   `/open/deviceCodeToToken`，雨雀 115yun/open 文档）。需 AppID、给 token 而非 cookie、
   最多 3 个并发授权、只能用 open API 面——webapi 生态用不了。现阶段坚持 cookie 流。
